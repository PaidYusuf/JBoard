Task Management System

System Analysis, Architecture Design, and Database Specification Document (V4)

This document is the definitive technical specification for a multi-tenant, role-based, and security-focused Web-Based Task Management Software with an optimized centralized logging infrastructure, developed based on initial handwritten wireframes.

1. User Roles and Interface Architecture

To operate the business logic and direct interface dynamics across the system, three fundamental roles and their respective dedicated panels have been defined:

1.1. SuperAdmin Dashboard

  • Global Dashboard: Tracking the total number of active groups (companies/teams), total users, and overall system capacity metrics.
  • Group / Company Management: The processes of adding, editing, or suspending new working groups in the system.
  • Plan Management (Plans): Defining and manipulating subscription packages (pricing and maximum user limits).
  • Centralized Audit (Audit Logs): High-level monitoring of logs generated by all groups and users across the system.

1.2. Admin Panel (Group / Team Manager)

  • Tasks of Team: The main screen listing the tasks of the members belonging only to the manager's group, featuring advanced filtering tools.
  • Gantt View: A visual interface mapping the start and deadline dates of tasks on a timeline.
  • Statistics Panel (Stats & Names Data): A component presenting graphical data on team members' performance metrics, completed tasks, and ongoing tasks.
  • Add Task / Assign Task: When creating a new task, the Admin determines the following parameters: User Select (group member), Task Name, Task Details, Start Date, and Deadline.

1.3. User Panel (End User / Employee)

  • Tasks List: The main screen where tasks assigned to the user are listed, differentiated by icons according to their process status (Not Started, In Progress, Completed).
  • Gantt / Date View: An integrated calendar structure where the user can organize their own schedule and task deadlines.
  • Task Details View (Click on Task): The page that opens when a task is clicked, containing: Details (task description), Task Report (text input area for the user's report), and Upload File (tool for uploading files like PDF, CSV, etc.).

2. Security, Session, and Logging Architecture

2.1. Cookie & Session Management (Secure Session)

After a successful user login, the system generates a cryptographically random SessionToken and writes it to the USER table alongside a SessionExpiresAt timestamp (default: 24 hours from login). This token is delivered to the browser as a Secure, HttpOnly cookie. On every subsequent request, the Backend validates both that the cookie value matches the stored token and that the current time is before SessionExpiresAt; if either check fails, the request is rejected with 401 Unauthorized and the user is redirected to the login page.

  • Token Rotation: On every successful login, the old SessionToken is discarded and a new one is generated. This prevents session fixation attacks.
  • Logout Invalidation: The logout action sets SessionToken = NULL and SessionExpiresAt = NULL in the database, immediately invalidating the session server-side even if the cookie somehow persists on the client.
  • Expiry Enforcement: Token expiration is checked on the Backend, not trusted from the client. The 24-hour window resets only on explicit re-login, not on activity.

2.2. Centralized Logging System (Logs)

For system security, bug tracking, and SuperAdmin auditing, all critical operations are written to a centralized table maintained in the relational database:

  • LoginLog: Successful/failed login attempts, login timestamps, and security authentication movements.
  • TaskLog: Task creation, deletion, status updates, report inputs, and file upload actions.

💡 Crucial Performance and Optimization Note: To prevent the logging operations from bloating the database and slowing down the system over time, two architectural rules must be implemented:
1. Asynchronous Logging: Logging operations should not block the main code execution flow; they must be handled in the background via a queue/thread mechanism.
2. Database Indexing: Indexes must be defined for the LogType and Timestamp fields in the LOGS table to optimize historical log queries.

3. Database Relational Schema Design

3.1. USER Table

Field Name Key Type Description / Business Logic
UserIDPKThe user's unique system identifier.
GroupIDFKThe group the user belongs to. Linked to the GROUP table. NULL for solo users until a personal group is auto-created.
EmailUNIQUEThe user's email address. Used as the login identifier. Must be unique across the entire system.
PasswordHash-The user's password stored as a secure hash (bcrypt or Argon2). Plain-text passwords must never be stored.
FName-First name of the user.
LName-Last name of the user.
Role-System authorization level (SuperAdmin, Admin, User).
AccountType-Distinguishes how the user registered: company (joined via invite code) or solo (self-registered, manages own tasks).
IsActive-Boolean flag. Set to false when a SuperAdmin suspends a user or group. Suspended users are blocked at session validation even with a valid token.
LastLogin-Timestamp of the user's last successful login.
SessionToken-Cryptographically random token for secure cookie verification. Set to NULL on logout.
SessionExpiresAt-Expiry timestamp for the current session token. Requests arriving after this time are rejected with 401. Set to NULL on logout.
CreatedAt-Timestamp of when the user account was created.

3.2. LOGS Table

Field Name Key Type Description / Business Logic
LogIDPKUnique log record identifier.
UserIDFKThe actor who performed the action. Linked to the USER table.
LogType-Log category (LoginLog, TaskLog, GroupLog, etc.).
Action-Details of the action performed (e.g., "Updated task status to completed").
Timestamp-Exact timestamp of when the action occurred (Indexed).

3.3. GROUP (Groups / Teams) Table

Field Name Key Type Description / Business Logic
GroupIDPKThe group's unique system identifier.
GroupName-The name of the group or company.
PlanIDFKThe subscription limit the group possesses. Linked to the PLANS table.
Type-Group category: company (managed by an Admin, invite-code based) or personal (auto-created for solo users).
Status-Operational state of the group: active or suspended. When suspended, all users in the group are blocked system-wide regardless of their individual IsActive flag.
CreatedAt-Timestamp of when the group was created in the system.

3.4. PLANS (Subscription Packages) Table

Field Name Key Type Description / Business Logic
PlanIDPKThe plan's unique system identifier.
PlanName-Display name of the plan (e.g., Free, Starter, Pro, Enterprise).
MaxUser-The maximum number of users a group utilizing this plan can add.
Price-The pricing value of the plan.

3.5. TASKS Table

Field Name Key Type Description / Business Logic
TaskIDPKThe task's unique system identifier.
UserIDFKThe user assigned to the task. Linked to the USER table.
GroupIDFKThe group the task belongs to. Linked to the GROUP table.
CreatedByFKThe Admin who created and assigned this task. Linked to the USER table.
TaskName-The short title of the task as set by the Admin during task creation.
TaskDetails-The full description and instructions for the task, written by the Admin.
TaskReport-The progress/completion report text submitted by the assigned User. NULL until the user submits a report.
Status-The current status of the task (Not Started, In Progress, Completed).
StartDate-The start date of the task.
EndDate-The final date by which the task must be delivered (Deadline).
CreatedAt-Timestamp of when the task was created.
UpdatedAt-Timestamp of the last modification to the task (status change, report update, etc.).

3.6. FILES (Uploaded Task Attachments) Table

Field Name Key Type Description / Business Logic
FileIDPKThe file record's unique system identifier.
TaskIDFKThe task this file belongs to. Linked to the TASKS table.
UserIDFKThe user who uploaded the file. Linked to the USER table.
StoredName-The UUID-based filename used on disk (e.g., f47ac10b-58cc.pdf). Never the original name, to prevent path traversal and collisions.
OriginalName-The original filename as provided by the user, stored for display purposes only.
MimeType-The verified MIME type of the file (e.g., application/pdf). Verified via Magic Bytes on upload, not trusted from the client.
FileSize-File size in bytes. Used for storage quota enforcement per group.
UploadedAt-Timestamp of when the file was uploaded.

3.7. INVITE_CODES (Group Invitation Tokens) Table

Used exclusively in the company registration flow. An Admin generates a one-time code for their group; a new user enters this code during registration to be linked to that group automatically.

Field Name Key Type Description / Business Logic
CodeIDPKThe invite code record's unique system identifier.
GroupIDFKThe group this code grants access to. Linked to the GROUP table.
CreatedByFKThe Admin who generated this code. Linked to the USER table.
CodeHash-A hashed version of the invite code. The plain-text code is only shown to the Admin once at generation time and never stored.
IsUsed-Boolean flag. Set to true once the code has been consumed by a registering user. One-time use only.
UsedByFKThe UserID of the user who consumed this code. NULL until redeemed. Linked to the USER table.
ExpiresAt-Expiry timestamp for the invite code (e.g., 72 hours after generation). Expired codes are rejected even if unused.
CreatedAt-Timestamp of when the invite code was generated.

4. File Storage and Directory Architecture (Storage Topology)

Documents uploaded by users via the interface are stored in a hierarchical and isolated directory structure on the server, maintaining security standards:

/root └── /GroupID # The isolated main folder of the respective company/group └── /UserID # The private folder of the user uploading the task └── /TaskID # The specific folder storing task-related documents (PDF, CSV, etc.)

5. Application Scenarios, Request Structures, and Security Specifications

The endpoint scenarios determined in accordance with the system's active operational states, HTTP request structures, and precautions against potential security vulnerabilities are configured below:

5.1. Admin Scenarios

Scenario 1: Assigning a New Task (Add Task / Assign Task)

  • Clicked Button: "Assign Task" (The submit button of the task creation form on the Admin Dashboard).
  • Button's Function: Takes the data entered into the form and creates a new task record in the database.
POST /api/admin/tasks Content-Type: application/json Cookie: SessionToken=abc123xyz... { "assignedUserId": 45, "taskName": "Extracting Database Schema", "taskDetails": "Determining relationships between tables", "startDate": "2026-06-20", "deadline": "2026-06-25" }
⚠️ Security Vulnerability Risk (IDOR - Insecure Direct Object Reference): A malicious Admin could write the ID of a user in another company/group into the assignedUserId parameter, assigning fake tasks to other groups.
🛡️ Solution: While processing the request, the Backend must verify the GroupID of the Admin making the request via the SessionToken. It must then be checked through the USER table whether the target user also possesses the same GroupID; if they do not match, the transaction must be rejected with a 403 Forbidden error.

Scenario 2: Filtering Team Tasks (Tasks of Team - Filter)

  • Clicked Button: "Filter" (To list tasks with an "In Progress" status).
  • Button's Function: Filters and displays the tasks belonging to the Admin's group according to the status parameter.
GET /api/admin/tasks?status=in_progress Cookie: SessionToken=abc123xyz...
⚠️ Security Vulnerability Risk (BOLA - Broken Object Level Authorization): If a parameter like ?groupId=2&status=in_progress is added to the URL on the client side, an attacker could change the groupId and leak internal tasks of other companies.
🛡️ Solution: The GroupID parameter must never be accepted from the client in the request. The Backend must assign which group's tasks are to be filtered directly from within by decoding the secure SessionToken.

Scenario 3: Viewing Group Statistics (Stats & Names Data)

  • Clicked Button: "View Statistics" (The graph/metrics button on the Dashboard).
  • Button's Function: Calculates the number of tasks completed and delayed by the personnel in the group and transmits this as graph data.
GET /api/admin/statistics Cookie: SessionToken=abc123xyz...
⚠️ Security Vulnerability Risk (DoS - Denial of Service & SQL Injection): Statistics calculations (COUNT, GROUP BY) are database-intensive operations. An attacker could lock the system by sending successive requests.
🛡️ Solution: "Rate Limiting" (e.g., maximum 10 requests per minute) must absolutely be applied to this endpoint. Parameterized Queries must be used for database queries instead of directly concatenating dynamic text.

5.2. User (Employee) Scenarios

Scenario 1: Updating Task Status (Tasks List - Status Change)

  • Clicked Button: "Mark as Completed" (The Status button on the task card).
  • Button's Function: Updates the status of the task assigned to the user in the TASKS table and writes to the LOGS table.
PATCH /api/user/tasks/1024/status Content-Type: application/json Cookie: SessionToken=xyz789abc... { "status": "completed" }
⚠️ Security Vulnerability Risk (IDOR): The user could manipulate another personnel's task status by changing the TaskID (1024) in the URL.
🛡️ Solution: On the Backend side, the user ID obtained from the SessionToken must exactly match the UserID value corresponding to the targeted TaskID in the database. A user must only be able to update their own work.

Scenario 2: Entering a Task Report (Task Report Input)

  • Clicked Button: "Save Report" (The button below the text area on the task details page).
  • Button's Function: Saves the text written by the user regarding the task into the task details in the database.
POST /api/user/tasks/1024/report Content-Type: application/json Cookie: SessionToken=xyz789abc... { "reportText": "Database relationships have been successfully established. Admin panel is ready." }
⚠️ Security Vulnerability Risk (XSS - Cross Site Scripting): The user could inject malicious JavaScript codes into the report area. When the Admin opens this report to read it, the code could execute in the browser.
🛡️ Solution: Textual data received from the user must undergo Input Sanitization before being written to the database and must be subjected to an HTML encode process when printed to the screen.

Scenario 3: Uploading a Task File (Upload File)

  • Clicked Button: "Upload File" (The Upload tool within the Task View).
  • Button's Function: Uploads the selected file to the isolated directory (/GroupID/UserID/TaskID) belonging to the relevant task on the server.
POST /api/user/tasks/1024/upload Content-Type: multipart/form-data; boundary=----WebKitFormBoundary... Cookie: SessionToken=xyz789abc... ------WebKitFormBoundary... Content-Disposition: form-data; name="file"; filename="report.pdf" Content-Type: application/pdf (File Byte Data) ------WebKitFormBoundary...--
⚠️ Security Vulnerability Risk (RCE - Remote Code Execution & Path Traversal): A user could upload malicious executable scripts (.php, .exe) to the server or step outside the system directory by manipulating the file name.
🛡️ Solution: The extension and MIME type of the incoming file must be verified on the server side (via Magic Bytes checking). The file name must not be saved as it comes from the user; it must be replaced with a random UUID in the background. Execute permissions for uploaded files must be disabled server-wide.

6. Registration and Onboarding Architecture

The system supports two distinct registration paths depending on whether the user belongs to an organization or is managing tasks independently.

6.1. Flow A — Company User Registration (Invite Code)

  1. An Admin navigates to their panel and generates a one-time invite code via POST /api/admin/invite-codes. The plain-text code is displayed once and immediately discarded server-side; only its hash is stored in the INVITE_CODES table.
  2. The Admin shares this code with the new team member out-of-band (e.g., via email or chat).
  3. The new user opens the Register page and enters the invite code. The frontend calls POST /api/auth/validate-invite-code.
  4. If the code is valid (not expired, not used), the Backend responds with the GroupName for that code. The UI displays: "You are joining [Company Name]. Is this correct?"
  5. The user confirms and a form appears requesting: First Name, Last Name, Email, and Password.
  6. On submission (POST /api/auth/register), the Backend creates the USER record with AccountType = company, links it to the correct GroupID, marks the invite code as IsUsed = true and sets UsedBy, then enforces the group's plan MaxUser limit before committing.
⚠️ Security Vulnerability Risk (Plan Limit Bypass): Without a server-side check, an attacker could send multiple simultaneous registration requests with the same code, exceeding the plan's user limit.
🛡️ Solution: The invite code must be marked as used and the user count must be checked within a single database transaction with a row-level lock to prevent race conditions.

6.2. Flow B — Solo User Registration (Self-Service)

  1. The user opens the Register page and selects "I'm registering for myself".
  2. A form appears requesting: First Name, Last Name, Email, and Password.
  3. On submission (POST /api/auth/register), the Backend:
    • Creates a new GROUP record with Type = personal and links it to the system's default free PlanID.
    • Creates the USER record with AccountType = solo, Role = Admin (they are their own manager), and links it to the newly created personal group.
  4. The user is redirected to their personal dashboard where they can create and manage their own tasks.

6.3. Login and Logout

  • Login: User submits Email and Password to POST /api/auth/login. Backend hashes the input and compares to PasswordHash. On success, a new SessionToken is generated, SessionExpiresAt is set to 24 hours from now, and a Secure HttpOnly cookie is returned.
  • Logout: User calls POST /api/auth/logout. Backend sets SessionToken = NULL and SessionExpiresAt = NULL for that user, then instructs the browser to clear the cookie.
